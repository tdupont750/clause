# Claude Container Project

This project builds and runs a Podman container for Claude Code CLI.

## Documentation

When changing any flag, option, or behavior in `clause`, always update both `CLAUDE.md` and `README.md` to reflect the change. The usage block in `README.md` should stay in sync with `./clause -h`.

## Project Structure

- `clause` — wrapper script that starts an ephemeral container session
- `default/` — profile template mirroring a real profile under `~/.clause/profiles/<name>/`; seeded into profiles on first use (every `default/<rel>` → `<profile>/<rel>`)
  - `default/Containerfile` — image definition (Ubuntu 24.04, Node.js 22, claude CLI, lazygit)
  - `default/args` — default `claude` args
  - `default/effort` holds the default effort level `max` (injected into the args at launch)
  - `default/.claude/settings.json` — default Claude settings
  - `default/.claude/CLAUDE.md` — default Claude instructions
  - `default/.claude/hooks/set-bg.sh` — terminal background-color hook (invoked by the seeded `settings.json` hooks)
  - `default/.claude.json` holds the empty Claude state `{}`
  - `default/.gitconfig` is an empty git config
  - `default/.claude/clause-sudo.log` is an empty sudo activity log
- `~/.clause/` — runtime state directory (auto-created on first run)
- `~/.clause/profiles/` — named profile directories, each with `.claude/`, `.claude.json`, and `.gitconfig`
- `~/.clause/profiles/default/` — built-in default profile (auto-created on first run)
- `<workspace>/.clause/`: per-workspace config dir holding the `profile` binding and optional `args`/`effort`/`mount` overrides; lives inside each workspace, so it travels with the folder

## Building

```bash
./clause image build
```

## Running

```bash
./clause [-w workspace] [-y] [-n] [-t] [-a <string>] [-e <level>]
./clause config [--profile] <key> [value]   # keys: args, effort, mount
./clause config [--profile] --unset <key>
./clause config --get <key>
./clause config --list                       # workspace + profile config, per scope
./clause profile create [name] | delete [name] | list
./clause image build | reset | suggest       # acts on the bound profile
./clause bind [profile] | --unset
./clause podman enable | disable | reset      # acts on the bound profile
./clause alias create | delete
./clause runtime <podman|docker> | --unset
./clause status
```

See `README.md` for full flag documentation.

## Key Decisions

- **Podman preferred, Docker supported** — `clause` auto-detects the container runtime at startup, preferring `podman` over `docker`; all container calls go through `$CONTAINER_CLI`; the override is stored in `~/.clause/runtime` and managed with `runtime <podman|docker>` / `runtime --unset` (see the `runtime` bind-style parsing decision below)
- **Ephemeral containers** — `--rm` removes the container on exit; all state is in bind mounts
- **No SSH** — sessions are interactive via the container runtime CLI (`podman run -it` or `docker run -it`)
- **Non-root user in container** — Claude runs as the in-image `claude` user (UID 1000); host UID is mapped via `--userns=keep-id:uid=1000,gid=1000` (podman) or `--user $(id -u):$(id -g)` (docker) so bind-mounted profile files stay writable. Passwordless `sudo` is available for ad-hoc installs; every sudo invocation is logged to `~/.clause/profiles/<name>/.claude/clause-sudo.log`
- **Profiles, not a single state dir** — each named profile under `~/.clause/profiles/` is independent; `default` is always bootstrapped
- **No auto-create for named profiles** — named profiles must be created explicitly with `profile create`; only `default` is created automatically on launch
- **Workspace binding file**: the workspace→profile binding is a single-line file `<workspace>/.clause/profile` naming the bound profile; read by `read_binding`, written by `write_binding`, removed by `remove_binding` (all act on the current `$WORKSPACE`, and reads are guarded so they never create anything). No central registry, so the binding travels with the folder when moved but is not enumerable across workspaces. This replaced `~/.clause/clause.conf` and its awk-based `config_get`/`config_set`/`config_del`/`config_del_by_profile` helpers
- **Bootstrap is lazy** — `~/.clause/` and `~/.clause/profiles/default/` (fully seeded from `default/` via `seed_profile`) are created idempotently the first time a non-read-only command runs (`bootstrap_state` no longer touches a central mapping file; `clause.conf` was removed); the pure read-only commands (`profile list` / `status`, `config --get` / `config --list`, and `-h`) never touch disk (the `bootstrap_state` allowlist is now `status|profile-list` plus `config` get/list)
- **`image build`, not bare container CLI** — image build is done via `clause image build`; the script errors with a clear message if the image is missing
- **Profile named only via `bind` / `profile create` / `profile delete`** — a profile name is never passed as a positional to launch or to `image`/`podman`; those act on the workspace's *bound* profile (`read_binding` → `default`). The only commands that take a profile name are `bind <profile>` (the session-side selector, `cmd_map`), `profile create <name>`, and `profile delete <name>` (registry management); all three set `OPT_PROFILE` from their own trailing token in `parse_args`/`parse_subcommand`. Launch/image/podman/status/config leave `OPT_PROFILE` empty, so `profile_for_management` resolves them via the workspace binding (`<workspace>/.clause/profile`). Removed in this change: the leading positional launch (`clause work`), the whole `run` command, and the `image`/`podman` trailing-profile arg; a leading bare word is now an `unknown command` error, and a stray trailing token on image/podman errors pointing at `bind`. `cmd_launch` no longer takes a requested profile: it uses the bound-or-default profile, keeping only the first-run "Save … → default?" y/n/q prompt for an unbound workspace (the old "override with '<requested>'?" branch is gone). Because no command selects a profile by a leading word, a profile may be named the same as any command word without collision (create/bind/delete take it as their trailing token; a bare `clause` launches it once bound)
- **One command per invocation** — parsing routes every command flag through `set_command` into a single `COMMAND` variable (default `launch`); a second command flag is a parse-time error naming both flags as typed, raised before any side effects. All top-level flow lives in functions: `main "$@"` on the script's last line runs `parse_args`, `resolve_workspace`, `bootstrap_state`, and `detect_runtime`, then dispatches command bodies (`cmd_*` functions) from a `case`; the read-only and needs-runtime allowlists are early-return `case` guards at the top of `bootstrap_state` / `detect_runtime`. Modifiers (`-a`, `-t`, `-w`, `-y`, `-n`) combine freely with any command
- **`profile create` seeds a fully-formed profile** — mirrors the repo's `default/` template into the profile via `seed_profile` (every `default/<rel>` → `<profile>/<rel>`, `cp -p` preserving the hook's executable bit; the empty runtime files `.claude.json` = `{}`, `.gitconfig`, and `.claude/clause-sudo.log` are tracked in `default/` too, so they seed the same way), and writes the current workspace's binding (`<workspace>/.clause/profile`; prompting before rebinding to a different profile, declining keeps the old binding). Launch no longer re-seeds: it validates the profile is complete via `require_profile_files` and errors on any missing file (see the defaults-live-in-`default/` bullet)
- **`profile delete` removes image and volume**: after deleting the profile directory, removes the `clause-<profile>` image and the `clause-<profile>-containers` volume if present. It cannot unbind other workspaces (bindings are local files, not enumerable), so a workspace still bound to the deleted profile hard-errors on its next launch until rebound; `cmd_delete_profile` prints a note saying so
- **Management-command naming, from `--<noun>-<verb>` flags to noun-verb subcommands (superseded by Tier B)**: management commands were historically long flags in noun-then-verb order (matching the newer podman/alias/runtime flag families), and four verb-noun stragglers were hard-renamed to conform (the profile create/delete and containerfile reset/suggest flags). Tier B then retired the management flags entirely, converting every one into a git-style noun-verb subcommand (`profile create`/`delete`/`list`, `image build`/`reset`/`suggest`, `map add`/`remove`/`list`/`show`, `podman enable`/`disable`/`reset`, `alias create`/`delete`, `runtime set`/`unset`, `status`), with no back-compat flag aliases. The internal `COMMAND` values and `cmd_*` function names carried over unchanged (the subcommands dispatch onto them). See the Noun-verb subcommands (Tier B) decision bullet below for the full surface and the clean-break list of removed flags
- **Per-profile `Containerfile`** — every profile has its own `Containerfile`; `image build` seeds missing profile files first and builds `clause-<profile>` from it (the image is always `clause-<profile>`, with no shared fallback image); `image reset` overwrites it with the current default
- **Claude args layering** — args appended to `claude` resolve in this order: (1) `-a/--args <string>` one-shot CLI override; (2) workspace-local `<workspace>/.clause/args` (present-file wins, even empty → no args); (3) profile-level `~/.clause/profiles/<name>/args` (likewise present-file wins; seeded default: `--dangerously-skip-permissions`, with the effort default moved to the sibling `effort` file); and, for the `default` profile only, when that profile `args` file is *absent*, a read-only fallback to the repo `default/args` template (via the `profile_tier_file` helper, source labeled `default template`) so `status`/`config` show the value a launch would seed (see the read-only default-template fallback bullet below). Resolution lives in `resolve_args`, a thin wrapper over the generic three-tier `resolve_layered` helper (mode `raw`: first line verbatim, a present file wins even if empty, no validation). Setting/clearing goes through the `config` subcommand: `config args <v>` (workspace) / `config --profile args <v>` (profile) share the `write_layered_file` helper; `config --unset args` (workspace `.clause/args`) / `config --profile --unset args` (profile `args`, resolves+requires the profile) share the `remove_layered_file` helper with the effort/mount keys. `config --unset args` *deletes* the workspace file so args fall through to the profile tier, distinct from `config args ''` which writes a present-but-empty file meaning "no args" (opt out). `config --get args` prints the raw effective args (including a one-shot `-a`) with no effort injection; `status` shows the post-injection launch line. All args are ignored under `-t`. See the git-style `config` subcommand decision below for the full surface
- **Effort override, layered and injected into args**: a dedicated effort knob resolves parallel to the args via `resolve_effort`, a wrapper over the same generic `resolve_layered` helper (mode `token`: whitespace-stripped single value, empty/whitespace skipped, validated by `validate_effort`): (1) `-e/--effort <level>` one-shot; (2) workspace `<workspace>/.clause/effort`; (3) profile `~/.clause/profiles/<name>/effort`; and, for the `default` profile only, when that profile `effort` file is *absent*, a read-only fallback to the repo `default/effort` template (via `profile_tier_file`, source `default template`; see the read-only default-template fallback bullet below). The resolved level is injected into the effective args by `apply_effort_to_args` (replaces an existing `--effort`/`--effort=` token, else appends), so the final `claude` command carries exactly one `--effort`. Valid levels are `low|medium|high|xhigh|max`, validated by `validate_effort`; `max` is allowed even though `settings.json`'s `effortLevel` rejects it, because this drives the CLI flag, not the setting. Setters/removers via `config`: `config effort <level>`/`config --unset effort` (workspace `.clause/effort`), `config --profile effort <level>`/`config --profile --unset effort` (profile `effort`); the setters share `write_layered_file` and the removers share `remove_layered_file` (which reports `No override at <path>` when absent) with the args knob. Precedence notes: a one-shot `-a` is a complete args override, so it bypasses the stored effort files too (only `-e` refines an `-a` line); an empty/whitespace effort file means "unset" and falls through (unlike `.clause/args`'s present-empty → "no args"); an invalid file value is warned about and skipped at launch. Injection lives in the claude-mode branch only, so `-t` bare runs are unaffected and keep reading `settings.json` `effortLevel`. `default/effort` = `max` is now seeded into every profile, so effort is a first-class, required file (part of `require_profile_files`) and a normal launch always injects `--effort max` unless a higher tier overrides it (the profile tier therefore also overrides a `--effort` written inside an `args` value; see the effort-first-class bullet). `config` set/unset need no runtime and are not read-only, so they pass through the existing `bootstrap_state`/`detect_runtime` guards unchanged (get/list are read-only). `status` prints the post-injection args plus the effort source; `config --get effort` prints the raw resolved level
- **`bind` binds a workspace to a profile (was the `map` noun)**: `bind [profile]` writes the workspace→profile binding (`<workspace>/.clause/profile`) without starting a session, warning and prompting if one already exists (`cmd_map`); `bind --unset` removes the current workspace's binding (`cmd_unmap`). `bind` is a top-level command word (like `status`), not a `<noun> <verb>` subcommand: it is parsed inline in `parse_args`, consuming an optional leading `--unset` (which selects COMMAND `unmap` over `map`) then an optional trailing bare-word profile (`clause bind work`; the old leading form `clause work bind` is gone with the positional), and rejecting a stray trailing token ("session options go before it"). It reuses the existing `map`/`unmap` COMMAND values and their `cmd_*` bodies, which now read and write the local binding file `<workspace>/.clause/profile` (via `read_binding`/`write_binding`/`remove_binding`) instead of the removed central `clause.conf`; both the CLI word and the storage changed from the old `map add`/`map remove`. `bind` is the sole session-side way to name a profile: launch takes no profile argument and uses the bound-or-default profile (see the profile-named-only-via-`bind` bullet). The reserved words are `config profile image podman alias runtime status bind` (`run` was removed); with the positional launch gone, a profile named like any of these is still fully usable, named via `profile create`/`bind` and launched with a bare `clause` once bound
- **`map show` folded into `status`, `map list` into `profile list`**: the two read-only `map` verbs were merged into the commands that already overlapped them, so the `map` noun could be retired whole. `status` (`cmd_status`) already printed the same `binding:` and `mount:` lines `map show` did, so `cmd_map_show` and the `map-show` COMMAND were deleted outright (no code moved). `profile list` (`cmd_profile_list`) now prints a `profiles:` section (all profiles, marking nested ones) followed by a `mappings:` section (the old `map list`/`cmd_list_all` body), so `cmd_list_all` and the `list-all` COMMAND were removed. (That `mappings:` section was itself later removed when `clause.conf` was dropped for local `<workspace>/.clause/profile` bindings, so `profile list` now lists profiles only; see the workspace-binding bullet.) Both stay read-only: nullglob yields nothing when there is no profiles dir, so neither creates `~/.clause`
- **`status` config dashboard**: `cmd_status` prints the full effective config for the current directory: profile, binding, mount (+ source), effective args (+ source), effective effort (+ source), runtime, and image-built status. Read-only: in the `bootstrap_state` allowlist (guard is now `status|profile-list`, plus a `config` arm that is read-only only for `get`/`list`), so it never creates `~/.clause`. Its `binding:` and `mount:` lines make `status` a superset of the removed `map show`. `config --list` is the config-only sibling, but with a **different job**: it dumps the *stored* config per scope (workspace tier + profile tier, read raw), while `status` resolves the single *effective* value and its origin. So `config --list` has no `--show-origin` (the origin question is answered by `status`); see the per-scope `config --list` bullet below. Resolves the profile via `profile_for_management` (not `require_profile`), so a missing profile degrades to `(not created)` instead of erroring (`resolve_args`/`resolve_effort` guard each read with `[[ -f ]]`, so an uncreated *named* profile dir just yields no profile-tier value; the `default` profile instead falls back to the `default/` template so its args/effort read as the post-seed launch values, see the read-only default-template fallback bullet below). Runtime/image use a soft probe local to `cmd_status` (read `RUNTIME_FILE`, else `command -v podman`/`docker`, else `(none found)`), deliberately bypassing the hard `detect_runtime` (which exits when no runtime is present) so status works on a host with no container runtime, hence NOT added to `detect_runtime`'s allowlist. Now a bare subcommand word (`status`), special-cased in the `parse_args` loop and dispatched like the other nouns, not a flag
- **Effective read-only views fall back to the `default/` template for the `default` profile**: `status` and `config --get` (the *effective*-value views) report the args/effort a launch would *actually* use, not just the raw on-disk profile files. (`config --list` is deliberately excluded: it is a per-scope *stored* view, so it shows the raw profile file, `(unset)` when absent; see the per-scope `config --list` bullet.) A launch always re-seeds the `default` profile from the repo `default/` template (`bootstrap_state` → `seed_profile`) *before* reading its `args`/`effort`, so an **absent** `default`-profile file resolves to the template value at launch; but read-only views must not seed. To close that gap, `resolve_args`/`resolve_effort` pick the profile-tier file through the `profile_tier_file` helper: for the `default` profile only, when `~/.clause/profiles/default/<key>` is absent (and the repo `default/<key>` exists), it substitutes the template path as the profile tier fed to `resolve_layered`, and the winning source is relabeled `default template`. This routes the template value through the exact same mode/validation as launch (raw for args, token+`validate_effort` for effort). Scope and edges: **default-profile only** (named profiles never re-seed at launch, they hard-error on an incomplete profile, so their read-only views still show `(no args)`/`(unset)`); **absent-only** (a present-but-empty profile file is a real value `seed_profile` never overwrites, so it is used as-is, matching launch, e.g. an `args` opt-out or an empty `effort` = "unset"); higher tiers (one-shot `-a`/`-e`, workspace `<workspace>/.clause/<key>`) still win over the template. **Launch stays byte-identical**: its `default`-profile files always exist post-seed, so `profile_tier_file`'s `! -f` guard is false and the template is never consulted at launch. `mount` has no template tier (nothing is seeded for it), so an un-pinned mount still reads `(unset)`. Implementation `set -e` note: the source-relabel in `resolve_args`/`resolve_effort` uses an `if [[ … ]]; then … fi` block, **not** a trailing `[[ … ]] && …` — as the function's last statement a false test would return nonzero and abort every caller (status, config, and *launch*) under `set -e`
- **Per-profile `.gitconfig`** — each profile has its own `.gitconfig` bind-mounted at `/home/claude/.gitconfig`; starts empty, persists across sessions
- **Encoded workspace mount** — the host workspace is bind-mounted at `/workspace/<encoded-host-path>` and `-w` sets the container cwd to the same path, where `<encoded-host-path>` is the host workspace with `/` and `.` replaced by `-` (same scheme Claude uses for `~/.claude/projects/`). This keeps Claude's per-project state (logs, todos, history) separated by host workspace even when multiple workspaces share a profile. The encoding is factored into the `encode_path` helper (same two substitutions, same order → default is byte-identical), and the *logical* path it encodes is resolved by `resolve_mount_path` so it can be pinned (see the mount override decision) to survive moving the host folder
- **Mount path override, layered (workspace-only, no profile tier)**: a workspace can pin the container-side mount path so Claude's path-keyed history survives moving the folder on the host. `resolve_mount_path` sets `MOUNT_VALUE` (the logical host path that `encode_path` turns into `CONTAINER_WORKSPACE`, the bind target + cwd) and `MOUNT_SOURCE`, with precedence: (1) `<workspace>/.clause/mount` (first line; `read -r` trims surrounding whitespace, keeps internal spaces; empty → ignored); (2) the real `$WORKSPACE` (default, unchanged). The one-shot `--mount <path>` launch flag was removed (clean break, no alias; `--mount` now errors as an unknown option), leaving the workspace file as the only override tier. The bind-mount *source* is always the real `$WORKSPACE`; only the container-side target/cwd are pinned. The `.clause/mount` file lives inside the workspace deliberately, so it travels with the folder when moved (the mechanism that keeps the pin in effect); the stored value is a logical absolute path, kept verbatim (dumb/predictable). `validate_mount_path` requires a leading `/` and rejects a trailing slash except root `/` (a trailing slash encodes to a stray `-` that no realpath produces, silently orphaning history); so pin the canonical path (`$(pwd -P)`); the file value is revalidated in `resolve_mount_path` (invalid → warn on stderr + fall through to `$WORKSPACE`, so a hand-edit cannot break launch). `resolve_mount_path`'s early returns are `return 0` (load-bearing under `set -e`: a bare `return` would propagate the failed `[[ -f ]]` test and abort the bare call on the common no-override path). The override changes container *layout*, not `claude` args, so it applies in both claude and `-t` bash modes (called outside the `CONTAINER_CMD` branch in `cmd_launch`, unlike args/effort). Set/clear via `config mount <path>` / `config --unset mount` share `write_layered_file` / `remove_layered_file` with the args and effort keys; there is no profile tier, so `config --profile mount` is a parse-time error (workspace-scoped only). Nothing is seeded in `default/` and no override file exists by default, so launch is byte-identical until an override is set; `config` set/unset need no runtime and are not read-only, so they pass through the existing `bootstrap_state`/`detect_runtime` guards unchanged. `status` reports the effective `mount:` path and its source, and `config --list` shows the raw workspace `mount` file (`(unset)` when not pinned) under its workspace scope only, since mount has no profile tier (mount is not a `claude` arg, so it never enters the args line)
- **Git-style `config` subcommand (Tier A)**: the three layered knobs (args/effort/mount) are managed through one git-shaped subcommand instead of per-knob flags. Grammar: `clause config [--profile|--workspace] [--get|--unset|--list] <key> [value]`, parsed by `parse_config_args` (Phase 1 consumes leading flags, stops at the first non-flag; Phase 2 takes a positional `<key> [value]` verbatim, so a dash-prefixed value like `config args '--effort high'` is never read as a flag). Keys are `args|effort|mount` (`validate_config_key`); value validation reuses `validate_effort` / `validate_mount_path` at parse time. Scope defaults to the workspace tier; `--profile` targets the profile tier, i.e. the workspace's *bound* profile (there is no positional profile selector; `--workspace` is the explicit default). `mount` is workspace-only, so `--profile mount` is a parse-time error. `config <key> <value>` sets, bare `config <key>` gets (implicit, git-style), `config --get` prints one raw effective value (scriptable, **no** effort-into-args injection, which stays a launch concern shown by `status`), `config --unset` clears, `config --list` dumps the stored config per scope (see the per-scope `config --list` bullet). `cmd_config` dispatches to `config_do_{set,unset,get,list}`; set/unset/get reuse the same `write_layered_file` / `remove_layered_file` / `resolve_args` / `resolve_effort` / `resolve_mount_path` helpers as launch, so precedence (workspace > profile) is identical (list instead reads each scope's file directly); profile-scope writes call `require_profile`, reads use `config_resolve_profile_silent` (degrades on a missing profile like `cmd_status`). `bootstrap_state` treats `config` as read-only **only** for `get`/`list` (never creates `~/.clause`); `set`/`unset` bootstrap like the old setters. Never needs a runtime. This **hard-replaced 12 flags with no aliases** (consistent with the noun-verb clean-break policy): `-A/--args-view`, `--args-set{,-profile}`, `--args-remove{,-profile}`, `--effort-set{,-profile}`, `--effort-remove{,-profile}`, `--mount-set`, `--mount-remove` all now error as unknown options. `config` is a reserved **command word** (a bare leading `config` is the subcommand); since the positional launch was removed, a profile may still be *named* `config` and launched with a bare `clause` once bound. Unchanged: `status` (the broad dashboard, the `git status` analog) and the one-shot launch flags `-a`/`-e` (git keeps `-c` alongside dedicated flags). Scope naming `--profile`/`--workspace` was chosen (by the user) over git-literal `--global`/`--local` because clause's tier is per-profile, not one user-wide file
- **`config --list` shows stored config per scope (not the resolved effective value); `--show-origin` removed**: `config --list` prints two labeled sections, `workspace (<workspace>/.clause)` and `profile <name> (<dir>)`, each listing that tier's keys read *directly* from its files with no cross-tier resolution: `args`/`effort` (and `mount` under workspace only, since mount has no profile tier). `config_do_list` no longer calls `resolve_args`/`resolve_effort`/`resolve_mount_path`; it reads each file via `config_scope_value` (absent → `(unset)`, present-but-empty first line → `(empty)`, else the first line verbatim) printed by `config_list_entry`. Consequences: (1) **no default-template fallback** here (unlike `status`/`config --get`), so an unseeded `default` profile key reads `(unset)`, honestly reflecting that nothing is stored yet, while `status` still shows the effective post-seed value + source (this is the intended `--list` vs `status` split, requested by the user); (2) **`--show-origin` was removed** (clean break: `--show-origin` now errors as an unknown config option, `OPT_CONFIG_SHOW_ORIGIN` and the old `config_list_line` deleted) because the scope headings already say where each value lives and the "which tier wins + exact source" question is `status`'s job; (3) scope flags (`--profile`/`--workspace`) are ignored by `--list` (it always prints both sections, since the default scope is already `workspace` and cannot be distinguished from an explicit one). Still read-only (in `bootstrap_state`'s `config` get/list arm; reads via `config_resolve_profile_silent`, so a missing profile degrades to a `(not created)` heading, never creating `~/.clause`). Supersedes the "`config --list` is the `git config --list` analog" framing in the Tier A bullet above
- **Noun-verb subcommands (Tier B)**: extending Tier A's `config`, every remaining management and info command became a git-style noun-verb subcommand, retiring the last management/info flags. Full surface: `config` (Tier A), `profile create|delete|list`, `image build|reset|suggest`, `podman enable|disable|reset`, `alias create|delete`, `runtime set <podman|docker>|unset`, `status`, and the top-level `bind` / `bind --unset`. The words `config profile image podman alias runtime status bind` are reserved (a bare word matching one is the subcommand). (This list originally also held `run`, the escape hatch for launching a command-named profile, and a leading positional selected a profile to launch; both were later removed, so profiles are named only via `bind` / `profile create` / `profile delete` and launch always uses the bound profile: see the profile-named-only-via-`bind` bullet.) Clean break, no flag aliases: the old `-s`, `-l`, `-L`, `-C`, `-D`, `-R`, `-S`, `-b`, `-P`, `-m`, `-u` short flags and their `--status`, `--list-all`, `--profile-create`, `--profile-delete`, `--containerfile-reset`, `--containerfile-suggest`, `--build`, `--podman-enable`, `--podman-disable`, `--podman-reset`, `--alias-create`, `--alias-delete`, `--runtime-set`, `--runtime-remove`, `--map`, `--unmap` long forms all now error as unknown options. Parser design: `parse_subcommand` maps each `<noun>.<verb>` to the SAME internal `COMMAND` value the old flags used (`image build`→`build`, `image reset`→`reset-containerfile`, `profile create`→`create-profile`, `runtime unset`→`runtime-remove`, and so on), so the `main` dispatch `case` and every `cmd_*` body are reused unchanged; `set_command` now takes a multi-word label (e.g. `image build`) purely for the conflict-error message. `status` is special-cased in the main `parse_args` loop rather than `parse_subcommand`, setting its command directly (`bind` is likewise parsed inline; `run` sat here too, consuming a verbatim profile name, before it was removed). The old combined list view was replaced by `profile list` (COMMAND `profile-list` → `cmd_profile_list`, listing profiles) and `status` (binding + mount); the old `cmd_list` was removed. (When the `map` noun was later retired, `map show` folded into `status` and `map list`/`cmd_list_all` into `profile list`, which grew a `mappings:` section, later dropped when `clause.conf` was removed; see the `bind` bullet.) Grammar: session options (`-t -w -a -e -y -n`) must precede the subcommand; a trailing token after `<noun> <verb>` is an optional profile name only for `profile create`/`delete` (every other subcommand acts on the bound profile and rejects one, erroring toward `bind`), and a required `podman|docker` value for `runtime set`; a stray dash-token after a subcommand is a clear error; a `-h`/`--help` verb prints usage (like `config`). `bootstrap_state`'s read-only allowlist is now `status|profile-list` (plus `config` get/list); `detect_runtime`'s allowlist is unchanged (`build|delete-profile|podman-reset|launch`). Tier A's `config` established this pattern
- **`runtime` is `bind`-style, not a `<noun> <verb>` subcommand (`runtime <podman|docker>` / `runtime --unset`)**: `runtime` was pulled out of the Tier B `parse_subcommand` noun group and is now parsed inline in `parse_args` exactly like `bind`: a top-level command word taking either a positional `podman|docker` value (COMMAND `runtime-set`) or `--unset` (COMMAND `runtime-remove`). This replaces the Tier B `runtime set <value>` / `runtime unset` verbs (clean break, no alias): `runtime set podman` now reads `set` as the value and errors on the trailing `podman`, and `runtime unset` reads `unset` as the value and fails `cmd_runtime_set`'s podman|docker validation. Edge cases mirror `bind`: bare `runtime` errors asking for a value or `--unset`; a dash token other than `--unset` is an unknown-option error; `runtime -h`/`--help` prints usage; a trailing token after the value or `--unset` errors ("session options go before it"). The value is still validated (podman|docker, present on PATH) in `cmd_runtime_set`, not at parse time. The internal COMMAND values (`runtime-set`, `runtime-remove`) and the `cmd_runtime_set` / `cmd_runtime_remove` bodies are unchanged, so `bootstrap_state` (falls through to `seed_profile`) and `detect_runtime` (hits the tolerant `*) return 0` arm, so runtime management still works on a runtime-less host) behave identically, as does the destructive `confirm_yes` "type 'yes'" gate on unset. `runtime` stays a reserved word. Consequently `subcommand_error` no longer lists a `runtime` noun, and `parse_subcommand`'s `runtime.set`/`runtime.unset` cases plus its `runtime-set` trailing-value branch were removed (that function now only sets `OPT_PROFILE`, never `OPT_RUNTIME_SET`). Supersedes the `runtime set`/`unset` portion of the Tier B surface above
- **Help groups commands by scope; `runtime`/`alias` under a `global` section**: `usage()` (and thus `-h`) lists the workspace- and profile-scoped commands (`config`, `bind`, `profile`, `image`, `podman`, `status`) under `commands (then exit):`, then the two user/host-wide setup commands under a separate `global (machine-wide setup):` heading: `runtime <podman|docker>` / `runtime --unset` (state in `~/.clause/runtime`) and `alias create|delete` (state in the shell rc files), both independent of any workspace or bound profile. Presentation only, no behavior/parse change. `profile` stays in the main list despite managing the global profile registry, because it is tied to the working context (it binds the workspace, and `image`/`podman`/`config --profile` act on a profile). `README.md`'s usage block mirrors this verbatim and must stay byte-identical to `./clause -h`
- **`default/` — profile template** — the repo's `default/` directory mirrors a profile's full layout (`Containerfile`, `args`, `effort`, `.claude.json`, `.gitconfig`, `.claude/settings.json`, `.claude/CLAUDE.md`, `.claude/hooks/set-bg.sh`, `.claude/clause-sudo.log`); `seed_profile` copies each file to the matching profile path on first use if missing, never overwriting afterward (users can freely modify their profile's copies). The formerly code-scaffolded runtime files (`.claude.json` = `{}`, empty `.gitconfig`, empty `.claude/clause-sudo.log`) are now tracked in `default/` too (the sudo log is force-added past the repo's `.claude/` gitignore), so `seed_profile` is the only seeding path and no code generates default file contents. Seeded copies are deleted only when the profile is deleted
- **`bypassPermissions` by default** — `default/.claude/settings.json` ships with `permissions.defaultMode = "bypassPermissions"`. Profiles created before this change keep whatever was already in their `settings.json`.
- **Official plugins enabled by default**: `default/.claude/settings.json` ships with an `enabledPlugins` map turning on two official plugins for every new profile, `skill-creator@claude-plugins-official` and `claude-md-management@claude-plugins-official`. The `claude-plugins-official` marketplace is baked into the CLI and auto-installs on launch (unless `CLAUDE_CODE_DISABLE_OFFICIAL_MARKETPLACE_AUTOINSTALL` is set), so no marketplace registration or plugin-cache seeding is needed; the plugins install on the profile's first session, which needs network access. Profiles created before this change keep whatever was already in their `settings.json`.
- **`effortLevel` defaults to xhigh**: `default/.claude/settings.json` ships with `effortLevel: "xhigh"`, Claude Code's settings key for startup reasoning effort (valid values `low`, `medium`, `high`, `xhigh`; the launch resolver reads `settings.effortLevel` only when no `--effort` flag is passed). Because normal launches pass `--effort` from the `effort` file (injected into the args) and that flag overrides the setting, `effortLevel` takes effect only for a bare `claude` run inside a `-t` terminal session. Deliberately kept in `settings.json` rather than raising the seeded `effort` file, so the launch default is unchanged. To override effort on a normal launch, use the layered effort override (the `resolve_effort` / `config effort` decision above), which injects `--effort` into the args; this `settings.json` value still governs only `-t` bare `claude` runs. Profiles created before this change keep whatever was already in their `settings.json`.
- **Nested podman is opt-in per profile**: `podman enable` creates the `~/.clause/profiles/<name>/nested` marker file and offers to append a managed block (between `# clause-nested-begin` / `# clause-nested-end`) to the profile's `Containerfile`; `podman disable` reverses both. The block's heredoc lives in the `clause` script, not in `default/`, so non-nested images stay lean and legacy profiles work the same way. Rebuild required after enabling.
- **Nested launch flags per runtime**: when the marker is present, launch adds `--device /dev/fuse --device /dev/net/tun --security-opt label=disable` plus the storage volume; docker additionally gets `seccomp=unconfined` and `apparmor=unconfined` (its default profiles block unshare/mount). Missing devices are skipped with a warning, and launch warns (non-fatally) if the profile `Containerfile` does not appear to install podman.
- **Per-profile nested storage volume**: inner podman storage lives in the named volume `clause-<profile>-containers` mounted at `/home/claude/.local/share/containers` (persists inner images across ephemeral sessions, allows native overlayfs, keeps nested-subuid-owned files out of the profile dir where `rm -rf` could not delete them). Removed automatically by `profile delete` and on demand by `podman reset` (typed confirmation); selective cleanup is `podman system prune` inside a session.
- **lazydocker in nested block**: the managed nested-podman block also installs the latest lazydocker release binary (same GitHub-redirect fetch as lazygit) and bakes `~/.config/lazydocker/config.yml` mapping compose actions to `podman-compose`. A `lazydocker` bash function (alias `ld`) appended to the container `.bashrc` starts `podman system service` on demand and points `DOCKER_HOST` at the socket. Profiles with the old block need `podman disable`, `podman enable`, and a rebuild to pick it up.
- **In-container `clause` alias**: `default/Containerfile` appends a `clause` alias to the container user's `.bashrc` that runs `claude --effort max --dangerously-skip-permissions` (mirrors the effective default launch: the profile's `args` plus its seeded `effort` of `max`; handy in `-t` terminal sessions). Baked into the image at build time: profiles with an existing `Containerfile` pick it up only after `image reset` (or adding the line manually) plus a rebuild, since seeding never overwrites a profile Containerfile.
- **lazygit in base image**: `default/Containerfile` installs the latest lazygit release binary into `/usr/local/bin` (arch-aware, fetched from GitHub at build time by following the `releases/latest` redirect) and adds an `lg` alias to the container user's `.bashrc`. Same caveat as the `clause` alias: existing profile Containerfiles pick it up only after `image reset` (or manual edit) plus a rebuild.
- **Workspace `.clause/` config dir + local binding, no central registry**: workspace-level config moved from loose dotfiles (`<workspace>/.clause-args`, `.clause-effort`, `.clause-mount`) into one `<workspace>/.clause/` directory (`args`, `effort`, `mount`), and the workspace→profile binding moved from the central `~/.clause/clause.conf` into `<workspace>/.clause/profile`. This deleted `clause.conf` and its `config_get`/`config_set`/`config_del`/`config_del_by_profile` helpers, replaced by `read_binding`/`write_binding`/`remove_binding` (which act on `$WORKSPACE`). The profile args file was renamed `clause-args` → `args` (in `default/` and each profile) so the workspace and profile knob filenames are symmetric (`args`, `effort`); `mount` and the `profile` binding stay workspace-only, and the resolvers (`resolve_args`/`resolve_effort`/`resolve_mount_path`/`config_file_for`) just point at the new paths. Motivation: the binding now travels with the folder (move-safe), the same reason `.clause/mount` already lives in the workspace, and the two-tier `resolve_layered` becomes truly symmetric. Tradeoffs (chosen deliberately, Option A): no migration or back-compat, so old `clause.conf` / `.clause-*` / `clause-args` files are simply no longer read; and bindings are no longer enumerable, so `profile list` lost its `mappings:` section (lists profiles only) and `profile delete` can no longer auto-unbind other workspaces (it prints a note; a stale binding hard-errors on that workspace's next launch). Kept unchanged: the first-run "Save … → default?" y/n/q prompt (now writes `<workspace>/.clause/profile`) and the hard error when a bound profile does not exist (`require_profile`). `write_layered_file` now creates the target's dir so the first workspace write creates `.clause/`; the read-only guarantees are unchanged (`status` / `profile list` / `config get` / `config list` still never create `~/.clause`, and `read_binding` is `[[ -f ]]`-guarded). This repo's own `.gitignore` now ignores `.clause/` (was `.clause-args`).
- **Workspace `.clause/` is self-ignoring (auto `.gitignore` of `*`)**: because the per-workspace `.clause/` dir lives inside the user's own project, creating it now also drops a `.gitignore` containing a single `*` so all of its contents (binding, args, effort, mount) are ignored by that project's repo automatically, with no manual step. The `ensure_workspace_config_dir` helper (`mkdir -p "$WORKSPACE/.clause"` then `[[ -f .gitignore ]] || printf '*\n' > .gitignore`) is the sole workspace-`.clause/` creation path: `write_binding` calls it, and `write_layered_file` routes only the workspace scope through it (a plain `[[ "$(dirname "$1")" == "$WORKSPACE/.clause" ]]` compare, so profile-tier writes under `~/.clause/profiles/<name>/` still just `mkdir -p` and get no `.gitignore`). Idempotent and non-destructive: the `.gitignore` is written only when absent, so it backfills a pre-existing `.clause/` on the next write and never overwrites a hand-edited one; `*` also ignores the `.gitignore` itself (the standard whole-folder-contents idiom). Read-only guarantees are unchanged (only the write paths `bind` / `config <key> <value>` / first-run save-binding / `profile create` reach it). Supersedes the README's old "you can add `.clause/` to your project's `.gitignore`" manual advice.
- **Profile defaults live in `default/`, not code; launch errors on an incomplete profile**: the runtime files that `init_profile_runtime` used to generate in code (`.claude.json` = `{}`, empty `.gitconfig`, empty `.claude/clause-sudo.log`) are now tracked files in `default/`, so `default/` is the single source of a profile's initial state and `seed_profile` is the only seeding path (`init_profile_runtime` was deleted). Seeding happens at bootstrap (full-seeds `default` from `default/`), `profile create`, and `image build`/`podman enable` (the explicit re-seed/repair paths). Launch no longer silently re-seeds: `cmd_launch` calls `require_profile_files` right after `require_profile`, which treats every `default/<rel>` as required and errors (listing the missing files, pointing at `clause image build`) if the bound profile lacks any. Rationale: `.claude.json` and `.gitconfig` are file bind-mounts that must pre-exist (else the runtime makes a directory in their place), and a strict gate surfaces a broken/pre-rename profile (e.g. one still holding `clause-args` and no `args`) instead of masking it. Read-only paths stay tolerant (`status` / `config --get` / `config --list` never call the gate and degrade to `(not created)`), and `default` never trips it because bootstrap re-seeds it. `require_profile_files` reuses the same `find "$SCRIPT_DIR/default" -type f` walk as `seed_profile`. `default/.claude/clause-sudo.log` is force-added past the repo's `.claude/` gitignore.
- **Effort is first-class, seeded in `default/effort` (not baked into `args`)**: the baseline effort used to live inside `default/args` as `--effort max`, a hidden default that made `config --get effort` / `status` read `(unset)` even though the effective effort was `max`. It now lives in its own `default/effort` = `max` file, and `default/args` is just `--dangerously-skip-permissions`. `resolve_effort` reads the profile `effort` tier and `apply_effort_to_args` injects it (an append now, since `args` no longer carries `--effort`), so the effective launch command is unchanged (`--dangerously-skip-permissions --effort max`, ordering aside). Consequences: `effort` joins the required set (`require_profile_files` now requires it; pre-existing profiles re-seed via `clause image build`), `config --get effort` and `status` report the real level, and because the effort ladder replaces any `--effort` in the resolved args, an `--effort` written inside an `args` value is overridden by the effort setting (at minimum the seeded profile `max`); set effort via `config effort`, not in `args`. `-t` bare runs are unaffected (`settings.json` `effortLevel` = `xhigh` still governs those).
