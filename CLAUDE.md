# Claude Container Project

This project builds and runs a Podman container for Claude Code CLI.

## Documentation

When changing any flag, option, or behavior in `clause`, always update both `CLAUDE.md` and `README.md` to reflect the change. The usage block in `README.md` should stay in sync with `./clause -h`.

## Project Structure

- `clause` — wrapper script that starts an ephemeral container session
- `default/` — profile template mirroring a real profile under `~/.clause/profiles/<name>/`; seeded into profiles on first use (every `default/<rel>` → `<profile>/<rel>`)
  - `default/Containerfile` — image definition (Ubuntu 24.04, Node.js 22, claude CLI, lazygit)
  - `default/clause-args` — default `claude` args
  - `default/.claude/settings.json` — default Claude settings
  - `default/.claude/CLAUDE.md` — default Claude instructions
  - `default/.claude/hooks/set-bg.sh` — terminal background-color hook (invoked by the seeded `settings.json` hooks)
- `~/.clause/` — runtime state directory (auto-created on first run)
- `~/.clause/profiles/` — named profile directories, each with `.claude/`, `.claude.json`, and `.gitconfig`
- `~/.clause/profiles/default/` — built-in default profile (auto-created on first run)
- `~/.clause/clause.conf` — workspace→profile mappings (auto-created on first run)

## Building

```bash
./clause --build
```

## Running

```bash
./clause [profile] [-w workspace] [--mount <path>] [-y] [-n] [-t] [-m] [-u] [-l] [-L] [-a <string>] [-e <level>]
./clause [profile] -s | --status
./clause [profile] -C | --profile-create
./clause [profile] -D | --profile-delete
./clause [profile] -R | --containerfile-reset
./clause [profile] -A | --args-view
./clause            --args-set <string>
./clause [profile]  --args-set-profile <string>
./clause            --args-remove
./clause [profile]  --args-remove-profile
./clause            --effort-set <level>
./clause [profile]  --effort-set-profile <level>
./clause            --effort-remove
./clause [profile]  --effort-remove-profile
./clause            --mount-set <path>
./clause            --mount-remove
./clause [profile] -S | --containerfile-suggest
./clause [profile] -P | --podman-enable
./clause [profile]  --podman-disable
./clause [profile]  --podman-reset
./clause --alias-create
./clause --alias-delete
./clause -b
```

See `README.md` for full flag documentation.

## Key Decisions

- **Podman preferred, Docker supported** — `clause` auto-detects the container runtime at startup, preferring `podman` over `docker`; all container calls go through `$CONTAINER_CLI`; the override is stored in `~/.clause/runtime` and managed with `--runtime-set` / `--runtime-remove`
- **Ephemeral containers** — `--rm` removes the container on exit; all state is in bind mounts
- **No SSH** — sessions are interactive via the container runtime CLI (`podman run -it` or `docker run -it`)
- **Non-root user in container** — Claude runs as the in-image `claude` user (UID 1000); host UID is mapped via `--userns=keep-id:uid=1000,gid=1000` (podman) or `--user $(id -u):$(id -g)` (docker) so bind-mounted profile files stay writable. Passwordless `sudo` is available for ad-hoc installs; every sudo invocation is logged to `~/.clause/profiles/<name>/.claude/clause-sudo.log`
- **Profiles, not a single state dir** — each named profile under `~/.clause/profiles/` is independent; `default` is always bootstrapped
- **No auto-create for named profiles** — named profiles must be created explicitly with `--profile-create`; only `default` is created automatically on launch
- **~/.clause/clause.conf format** — one `absolute-path=profilename` entry per line; parsed with awk for literal-safe matching
- **Bootstrap is lazy** — `~/.clause/`, `~/.clause/profiles/default/`, and `~/.clause/clause.conf` are created idempotently the first time a non-read-only command runs; `-l` / `-L` / `-A` / `-s` and `-h` are pure read-only and never touch disk
- **`--build` flag, not bare container CLI** — image build is done via `clause --build`; the script errors with a clear message if the image is missing
- **Positional profile argument** — profile is passed as a positional arg (e.g. `clause myprofile`), not `-p`; defaults to `default`
- **One command per invocation** — parsing routes every command flag through `set_command` into a single `COMMAND` variable (default `launch`); a second command flag is a parse-time error naming both flags as typed, raised before any side effects. All top-level flow lives in functions: `main "$@"` on the script's last line runs `parse_args`, `resolve_workspace`, `bootstrap_state`, and `detect_runtime`, then dispatches command bodies (`cmd_*` functions) from a `case`; the read-only and needs-runtime allowlists are early-return `case` guards at the top of `bootstrap_state` / `detect_runtime`. Modifiers (`-a`, `-t`, `-w`, `-y`, `-n`, positional profile) combine freely with any command
- **`--profile-create` seeds a fully-formed profile** — mirrors the repo's `default/` template into the profile via `seed_profile` (every `default/<rel>` → `<profile>/<rel>`, `cp -p` preserving the hook's executable bit), scaffolds the empty runtime files (`.claude.json` = `{}`, `.gitconfig`, `.claude/clause-sudo.log`), and adds the current workspace→profile mapping (prompting before overwriting a mapping to a different profile; declining keeps the old mapping). The launch path's "Init profile runtime data" block calls the same `seed_profile` as an idempotent lazy fallback, so profiles created before a template change still gain any missing files on next use
- **`--profile-delete` auto-unmaps and removes image** — after deleting a profile directory, removes all its workspace mappings and the `clause-<profile>` image if present
- **Management commands use `--<noun>-<verb>` order** — every management command's long flag reads noun-then-verb, matching the newer families (`--args-set`, `--podman-enable`, `--alias-create`, `--runtime-set`); the four verb-noun stragglers were hard-renamed to conform: `--create-profile`→`--profile-create`, `--delete-profile`→`--profile-delete`, `--reset-containerfile`→`--containerfile-reset`, `--suggest-updates`→`--containerfile-suggest`. Short flags (`-C`/`-D`/`-R`/`-S`), behavior, and the internal `COMMAND` values / `cmd_*` function names are unchanged, and there are NO back-compat aliases (the old verb-noun long names now error). The idiomatic verb-only commands `--map`/`--unmap`/`--list`/`--list-all`/`--build` are deliberately kept as-is
- **Per-profile `Containerfile`** — every profile has its own `Containerfile`; `--build` seeds missing profile files first and builds `clause-<profile>` from it (the image is always `clause-<profile>`, with no shared fallback image); `--containerfile-reset` overwrites it with the current default
- **Claude args layering** — args appended to `claude` resolve in this order: (1) `-a/--args <string>` one-shot CLI override; (2) workspace-local `<workspace>/.clause-args` (present-file wins, even empty → no args); (3) profile-level `~/.clause/profiles/<name>/clause-args` (likewise present-file wins; seeded default: `--effort max --dangerously-skip-permissions`). Resolution lives in `resolve_args`, a thin wrapper over the generic three-tier `resolve_layered` helper (mode `raw`: first line verbatim, a present file wins even if empty, no validation). Setters `--args-set` (workspace) and `--args-set-profile` (profile) share the `write_layered_file` helper; removers `--args-remove` (workspace `.clause-args`, `cmd_args_remove`) and `--args-remove-profile` (profile `clause-args`, `cmd_args_remove_profile`, calls `require_profile` first) share the `remove_layered_file` helper with the effort/mount removers and exit after (like `cmd_effort_remove`). `--args-remove` *deletes* the workspace file so args fall through to the profile tier, distinct from `--args-set ''` which writes a present-but-empty file meaning "no args" (opt out); both removers mutate disk (not read-only, like `--effort-remove`) and need no runtime. `-A/--args-view` prints the effective args (including `-a`) and reports the source. All args are ignored under `-t`
- **Effort override, layered and injected into args**: a dedicated effort knob resolves parallel to the args via `resolve_effort`, a wrapper over the same generic `resolve_layered` helper (mode `token`: whitespace-stripped single value, empty/whitespace skipped, validated by `validate_effort`): (1) `-e/--effort <level>` one-shot; (2) workspace `<workspace>/.clause-effort`; (3) profile `~/.clause/profiles/<name>/effort`. The resolved level is injected into the effective args by `apply_effort_to_args` (replaces an existing `--effort`/`--effort=` token, else appends), so the final `claude` command carries exactly one `--effort`. Valid levels are `low|medium|high|xhigh|max`, validated by `validate_effort`; `max` is allowed even though `settings.json`'s `effortLevel` rejects it, because this drives the CLI flag, not the setting. Setters/removers: `--effort-set`/`--effort-remove` (workspace `.clause-effort`), `--effort-set-profile`/`--effort-remove-profile` (profile `effort`); the setters share `write_layered_file` and the removers share `remove_layered_file` (which reports `No override at <path>` when absent) with the args knob. Precedence notes: a one-shot `-a` is a complete args override, so it bypasses the stored effort files too (only `-e` refines an `-a` line); an empty/whitespace effort file means "unset" and falls through (unlike `.clause-args`'s present-empty → "no args"); an invalid file value is warned about and skipped at launch. Injection lives in the claude-mode branch only, so `-t` bare runs are unaffected and keep reading `settings.json` `effortLevel`. Nothing is seeded in `default/` and no override file exists by default, so launch behavior is byte-identical until an override is set; the four commands need no runtime and are not read-only, so they pass through the existing `bootstrap_state`/`detect_runtime` guards unchanged. `-A/--args-view` prints the post-injection args plus the effort source
- **`-m`/`--map` for explicit mapping** — adds a workspace→profile mapping without starting a session; warns and prompts if a mapping already exists. Removed counterpart is `-u`/`--unmap`
- **`-l`/`--list` combined view** — prints the current workspace mapping and lists all profiles
- **`-L`/`--list-all` for every mapping** — prints the full contents of `~/.clause/clause.conf`
- **`-s`/`--status` config dashboard** — `cmd_status` prints the full effective config for the current directory: profile, mapping, mount (+ source), effective args (+ source), effective effort (+ source), runtime, and image-built status. Read-only: added to the `bootstrap_state` allowlist (guard is now `list|list-all|args-view|status`), so it never creates `~/.clause`. Resolves the profile via `profile_for_management` (not `require_profile`), so a missing profile degrades to `(not created)` instead of erroring (`resolve_args`/`resolve_effort` guard each read with `[[ -f ]]`, so an uncreated profile dir just yields no profile-tier value). Runtime/image use a soft probe local to `cmd_status` (read `RUNTIME_FILE`, else `command -v podman`/`docker`, else `(none found)`), deliberately bypassing the hard `detect_runtime` (which exits when no runtime is present) so status works on a host with no container runtime, hence NOT added to `detect_runtime`'s allowlist. A flag (`-s`/`--status`), not a bare `status` word (the positional is reserved for the profile name); the `-s`/`-S` case split mirrors `-l`/`-L`
- **Per-profile `.gitconfig`** — each profile has its own `.gitconfig` bind-mounted at `/home/claude/.gitconfig`; starts empty, persists across sessions
- **Encoded workspace mount** — the host workspace is bind-mounted at `/workspace/<encoded-host-path>` and `-w` sets the container cwd to the same path, where `<encoded-host-path>` is the host workspace with `/` and `.` replaced by `-` (same scheme Claude uses for `~/.claude/projects/`). This keeps Claude's per-project state (logs, todos, history) separated by host workspace even when multiple workspaces share a profile. The encoding is factored into the `encode_path` helper (same two substitutions, same order → default is byte-identical), and the *logical* path it encodes is resolved by `resolve_mount_path` so it can be pinned (see the mount override decision) to survive moving the host folder
- **Mount path override, layered (workspace + one-shot, no profile tier)**: a workspace can pin the container-side mount path so Claude's path-keyed history survives moving the folder on the host. `resolve_mount_path` sets `MOUNT_VALUE` (the logical host path that `encode_path` turns into `CONTAINER_WORKSPACE`, the bind target + cwd) and `MOUNT_SOURCE`, with precedence: (1) `--mount <path>` one-shot; (2) `<workspace>/.clause-mount` (first line; `read -r` trims surrounding whitespace, keeps internal spaces; empty → ignored); (3) the real `$WORKSPACE` (default, unchanged). The bind-mount *source* is always the real `$WORKSPACE`; only the container-side target/cwd are pinned. The `.clause-mount` file lives inside the workspace deliberately, so it travels with the folder when moved (the mechanism that keeps the pin in effect); the stored value is a logical absolute path, kept verbatim (dumb/predictable). `validate_mount_path` requires a leading `/` and rejects a trailing slash except root `/` (a trailing slash encodes to a stray `-` that no realpath produces, silently orphaning history); so pin the canonical path (`$(pwd -P)`); the one-shot is validated at parse time (hard error), the file value is revalidated in `resolve_mount_path` (invalid → warn on stderr + fall through to `$WORKSPACE`, so a hand-edit cannot break launch). `resolve_mount_path`'s early returns are `return 0` (load-bearing under `set -e`: a bare `return` would propagate the failed `[[ -f ]]` test and abort the bare call on the common no-override path). The override changes container *layout*, not `claude` args, so it applies in both claude and `-t` bash modes (called outside the `CONTAINER_CMD` branch in `cmd_launch`, unlike args/effort). Setters `--mount-set` (workspace `.clause-mount`) / `--mount-remove` share `write_layered_file` / `remove_layered_file` with the args and effort knobs; there is no profile tier and `cmd_mount_set` does not `require_profile` (workspace-scoped, like `cmd_args_set`). Nothing is seeded in `default/` and no override file exists by default, so launch is byte-identical until an override is set; the two commands need no runtime and are not read-only, so they pass through the existing `bootstrap_state`/`detect_runtime` guards unchanged (like `--effort-set`). `-l/--list` reports the effective `mount:` path and its source; `-A/--args-view` is untouched (mount is not a `claude` arg)
- **`default/` — profile template** — the repo's `default/` directory mirrors a profile's layout (`Containerfile`, `clause-args`, `.claude/settings.json`, `.claude/CLAUDE.md`, `.claude/hooks/set-bg.sh`); `seed_profile` copies each file to the matching profile path on first use if missing, never overwriting afterward (users can freely modify their profile's copies). Empty runtime files (`.claude.json`, `.gitconfig`, `.claude/clause-sudo.log`) are scaffolded by the script, not tracked in `default/`. Seeded copies are deleted only when the profile is deleted
- **`bypassPermissions` by default** — `default/.claude/settings.json` ships with `permissions.defaultMode = "bypassPermissions"`. Profiles created before this change keep whatever was already in their `settings.json`.
- **Official plugins enabled by default**: `default/.claude/settings.json` ships with an `enabledPlugins` map turning on two official plugins for every new profile, `skill-creator@claude-plugins-official` and `claude-md-management@claude-plugins-official`. The `claude-plugins-official` marketplace is baked into the CLI and auto-installs on launch (unless `CLAUDE_CODE_DISABLE_OFFICIAL_MARKETPLACE_AUTOINSTALL` is set), so no marketplace registration or plugin-cache seeding is needed; the plugins install on the profile's first session, which needs network access. Profiles created before this change keep whatever was already in their `settings.json`.
- **`effortLevel` defaults to xhigh**: `default/.claude/settings.json` ships with `effortLevel: "xhigh"`, Claude Code's settings key for startup reasoning effort (valid values `low`, `medium`, `high`, `xhigh`; the launch resolver reads `settings.effortLevel` only when no `--effort` flag is passed). Because normal launches pass `--effort` from `clause-args` and that flag overrides the setting, `effortLevel` takes effect only for a bare `claude` run inside a `-t` terminal session. Deliberately kept in `settings.json` rather than raising the `clause-args` effort, so the launch default is unchanged. To override effort on a normal launch, use the layered effort override (the `resolve_effort` / `--effort-set*` decision above), which injects `--effort` into the args; this `settings.json` value still governs only `-t` bare `claude` runs. Profiles created before this change keep whatever was already in their `settings.json`.
- **Nested podman is opt-in per profile**: `--podman-enable` creates the `~/.clause/profiles/<name>/nested` marker file and offers to append a managed block (between `# clause-nested-begin` / `# clause-nested-end`) to the profile's `Containerfile`; `--podman-disable` reverses both. The block's heredoc lives in the `clause` script, not in `default/`, so non-nested images stay lean and legacy profiles work the same way. Rebuild required after enabling.
- **Nested launch flags per runtime**: when the marker is present, launch adds `--device /dev/fuse --device /dev/net/tun --security-opt label=disable` plus the storage volume; docker additionally gets `seccomp=unconfined` and `apparmor=unconfined` (its default profiles block unshare/mount). Missing devices are skipped with a warning, and launch warns (non-fatally) if the profile `Containerfile` does not appear to install podman.
- **Per-profile nested storage volume**: inner podman storage lives in the named volume `clause-<profile>-containers` mounted at `/home/claude/.local/share/containers` (persists inner images across ephemeral sessions, allows native overlayfs, keeps nested-subuid-owned files out of the profile dir where `rm -rf` could not delete them). Removed automatically by `--profile-delete` and on demand by `--podman-reset` (typed confirmation); selective cleanup is `podman system prune` inside a session.
- **lazydocker in nested block**: the managed nested-podman block also installs the latest lazydocker release binary (same GitHub-redirect fetch as lazygit) and bakes `~/.config/lazydocker/config.yml` mapping compose actions to `podman-compose`. A `lazydocker` bash function (alias `ld`) appended to the container `.bashrc` starts `podman system service` on demand and points `DOCKER_HOST` at the socket. Profiles with the old block need `--podman-disable`, `--podman-enable`, and a rebuild to pick it up.
- **In-container `clause` alias**: `default/Containerfile` appends a `clause` alias to the container user's `.bashrc` that runs `claude --effort max --dangerously-skip-permissions` (mirrors the seeded `clause-args` default; handy in `-t` terminal sessions). Baked into the image at build time: profiles with an existing `Containerfile` pick it up only after `--containerfile-reset` (or adding the line manually) plus a rebuild, since seeding never overwrites a profile Containerfile.
- **lazygit in base image**: `default/Containerfile` installs the latest lazygit release binary into `/usr/local/bin` (arch-aware, fetched from GitHub at build time by following the `releases/latest` redirect) and adds an `lg` alias to the container user's `.bashrc`. Same caveat as the `clause` alias: existing profile Containerfiles pick it up only after `--containerfile-reset` (or manual edit) plus a rebuild.
