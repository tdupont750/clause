# Claude Container Project

This project builds and runs a Podman container for Claude Code CLI.

## Documentation

When changing any flag, option, or behavior in `clause`, always update both `CLAUDE.md` and `README.md` to reflect the change. The usage block in `README.md` should stay in sync with `./clause -h`.

## Project Structure

- `clause` â€” wrapper script that starts an ephemeral container session
- `default/` â€” profile template mirroring a real profile under `~/.clause/profiles/<name>/`; seeded into profiles on first use (every `default/<rel>` â†’ `<profile>/<rel>`)
  - `default/Containerfile` â€” image definition (Ubuntu 24.04, Node.js 22, claude CLI, lazygit)
  - `default/clause-args` â€” default `claude` args
  - `default/.claude/settings.json` â€” default Claude settings
  - `default/.claude/CLAUDE.md` â€” default Claude instructions
  - `default/.claude/hooks/set-bg.sh` â€” terminal background-color hook (invoked by the seeded `settings.json` hooks)
- `~/.clause/` â€” runtime state directory (auto-created on first run)
- `~/.clause/profiles/` â€” named profile directories, each with `.claude/`, `.claude.json`, and `.gitconfig`
- `~/.clause/profiles/default/` â€” built-in default profile (auto-created on first run)
- `~/.clause/clause.conf` â€” workspaceâ†’profile mappings (auto-created on first run)

## Building

```bash
./clause image build
```

## Running

```bash
./clause [profile] [-w workspace] [--mount <path>] [-y] [-n] [-t] [-a <string>] [-e <level>]
./clause [profile]  config [--profile] <key> [value]   # keys: args, effort, mount
./clause [profile]  config [--profile] --unset <key>
./clause [profile]  config --get <key>
./clause [profile]  config --list [--show-origin]
./clause [profile]  profile create [name] | delete [name] | list
./clause [profile]  image build [profile] | reset [profile] | suggest [profile]
./clause [profile]  bind [profile] | --unset
./clause [profile]  podman enable [profile] | disable [profile] | reset [profile]
./clause            alias create | delete
./clause            runtime set <podman|docker> | unset
./clause [profile]  status
./clause            run [profile]
```

See `README.md` for full flag documentation.

## Key Decisions

- **Podman preferred, Docker supported** â€” `clause` auto-detects the container runtime at startup, preferring `podman` over `docker`; all container calls go through `$CONTAINER_CLI`; the override is stored in `~/.clause/runtime` and managed with `runtime set` / `runtime unset`
- **Ephemeral containers** â€” `--rm` removes the container on exit; all state is in bind mounts
- **No SSH** â€” sessions are interactive via the container runtime CLI (`podman run -it` or `docker run -it`)
- **Non-root user in container** â€” Claude runs as the in-image `claude` user (UID 1000); host UID is mapped via `--userns=keep-id:uid=1000,gid=1000` (podman) or `--user $(id -u):$(id -g)` (docker) so bind-mounted profile files stay writable. Passwordless `sudo` is available for ad-hoc installs; every sudo invocation is logged to `~/.clause/profiles/<name>/.claude/clause-sudo.log`
- **Profiles, not a single state dir** â€” each named profile under `~/.clause/profiles/` is independent; `default` is always bootstrapped
- **No auto-create for named profiles** â€” named profiles must be created explicitly with `profile create`; only `default` is created automatically on launch
- **~/.clause/clause.conf format** â€” one `absolute-path=profilename` entry per line; parsed with awk for literal-safe matching
- **Bootstrap is lazy** â€” `~/.clause/`, `~/.clause/profiles/default/`, and `~/.clause/clause.conf` are created idempotently the first time a non-read-only command runs; the pure read-only commands (`profile list` / `status`, `config --get` / `config --list`, and `-h`) never touch disk (the `bootstrap_state` allowlist is now `status|profile-list` plus `config` get/list)
- **`image build`, not bare container CLI** â€” image build is done via `clause image build`; the script errors with a clear message if the image is missing
- **Positional profile argument** â€” profile is passed as a positional arg (e.g. `clause myprofile`), not `-p`; defaults to `default`
- **One command per invocation** â€” parsing routes every command flag through `set_command` into a single `COMMAND` variable (default `launch`); a second command flag is a parse-time error naming both flags as typed, raised before any side effects. All top-level flow lives in functions: `main "$@"` on the script's last line runs `parse_args`, `resolve_workspace`, `bootstrap_state`, and `detect_runtime`, then dispatches command bodies (`cmd_*` functions) from a `case`; the read-only and needs-runtime allowlists are early-return `case` guards at the top of `bootstrap_state` / `detect_runtime`. Modifiers (`-a`, `-t`, `-w`, `-y`, `-n`, positional profile) combine freely with any command
- **`profile create` seeds a fully-formed profile** â€” mirrors the repo's `default/` template into the profile via `seed_profile` (every `default/<rel>` â†’ `<profile>/<rel>`, `cp -p` preserving the hook's executable bit), scaffolds the empty runtime files (`.claude.json` = `{}`, `.gitconfig`, `.claude/clause-sudo.log`), and adds the current workspaceâ†’profile mapping (prompting before overwriting a mapping to a different profile; declining keeps the old mapping). The launch path's "Init profile runtime data" block calls the same `seed_profile` as an idempotent lazy fallback, so profiles created before a template change still gain any missing files on next use
- **`profile delete` auto-unmaps and removes image** â€” after deleting a profile directory, removes all its workspace mappings and the `clause-<profile>` image if present
- **Management-command naming, from `--<noun>-<verb>` flags to noun-verb subcommands (superseded by Tier B)**: management commands were historically long flags in noun-then-verb order (matching the newer podman/alias/runtime flag families), and four verb-noun stragglers were hard-renamed to conform (the profile create/delete and containerfile reset/suggest flags). Tier B then retired the management flags entirely, converting every one into a git-style noun-verb subcommand (`profile create`/`delete`/`list`, `image build`/`reset`/`suggest`, `map add`/`remove`/`list`/`show`, `podman enable`/`disable`/`reset`, `alias create`/`delete`, `runtime set`/`unset`, `status`), with no back-compat flag aliases. The internal `COMMAND` values and `cmd_*` function names carried over unchanged (the subcommands dispatch onto them). See the Noun-verb subcommands (Tier B) decision bullet below for the full surface and the clean-break list of removed flags
- **Per-profile `Containerfile`** â€” every profile has its own `Containerfile`; `image build` seeds missing profile files first and builds `clause-<profile>` from it (the image is always `clause-<profile>`, with no shared fallback image); `image reset` overwrites it with the current default
- **Claude args layering** â€” args appended to `claude` resolve in this order: (1) `-a/--args <string>` one-shot CLI override; (2) workspace-local `<workspace>/.clause-args` (present-file wins, even empty â†’ no args); (3) profile-level `~/.clause/profiles/<name>/clause-args` (likewise present-file wins; seeded default: `--effort max --dangerously-skip-permissions`). Resolution lives in `resolve_args`, a thin wrapper over the generic three-tier `resolve_layered` helper (mode `raw`: first line verbatim, a present file wins even if empty, no validation). Setting/clearing goes through the `config` subcommand: `config args <v>` (workspace) / `config --profile args <v>` (profile) share the `write_layered_file` helper; `config --unset args` (workspace `.clause-args`) / `config --profile --unset args` (profile `clause-args`, resolves+requires the profile) share the `remove_layered_file` helper with the effort/mount keys. `config --unset args` *deletes* the workspace file so args fall through to the profile tier, distinct from `config args ''` which writes a present-but-empty file meaning "no args" (opt out). `config --get args` prints the raw effective args (including a one-shot `-a`) with no effort injection; `status` shows the post-injection launch line. All args are ignored under `-t`. See the git-style `config` subcommand decision below for the full surface
- **Effort override, layered and injected into args**: a dedicated effort knob resolves parallel to the args via `resolve_effort`, a wrapper over the same generic `resolve_layered` helper (mode `token`: whitespace-stripped single value, empty/whitespace skipped, validated by `validate_effort`): (1) `-e/--effort <level>` one-shot; (2) workspace `<workspace>/.clause-effort`; (3) profile `~/.clause/profiles/<name>/effort`. The resolved level is injected into the effective args by `apply_effort_to_args` (replaces an existing `--effort`/`--effort=` token, else appends), so the final `claude` command carries exactly one `--effort`. Valid levels are `low|medium|high|xhigh|max`, validated by `validate_effort`; `max` is allowed even though `settings.json`'s `effortLevel` rejects it, because this drives the CLI flag, not the setting. Setters/removers via `config`: `config effort <level>`/`config --unset effort` (workspace `.clause-effort`), `config --profile effort <level>`/`config --profile --unset effort` (profile `effort`); the setters share `write_layered_file` and the removers share `remove_layered_file` (which reports `No override at <path>` when absent) with the args knob. Precedence notes: a one-shot `-a` is a complete args override, so it bypasses the stored effort files too (only `-e` refines an `-a` line); an empty/whitespace effort file means "unset" and falls through (unlike `.clause-args`'s present-empty â†’ "no args"); an invalid file value is warned about and skipped at launch. Injection lives in the claude-mode branch only, so `-t` bare runs are unaffected and keep reading `settings.json` `effortLevel`. Nothing is seeded in `default/` and no override file exists by default, so launch behavior is byte-identical until an override is set; `config` set/unset need no runtime and are not read-only, so they pass through the existing `bootstrap_state`/`detect_runtime` guards unchanged (get/list are read-only). `status` prints the post-injection args plus the effort source; `config --get effort` prints the raw resolved level
- **`bind` maps a workspace to a profile (was the `map` noun)**: `bind [profile]` adds a workspaceâ†’profile mapping without starting a session, warning and prompting if one already exists (`cmd_map`); `bind --unset` removes the current workspace's mapping (`cmd_unmap`). `bind` is a top-level command word (like `status`/`run`), not a `<noun> <verb>` subcommand: it is parsed inline in `parse_args`, consuming an optional leading `--unset` (which selects COMMAND `unmap` over `map`) then an optional leading-or-trailing bare-word profile (so `clause bind work` == `clause work bind`), and rejecting a stray trailing token ("session options go before it"). It reuses the existing `map`/`unmap` COMMAND values and their `cmd_*` bodies verbatim, so mapping behavior is byte-identical to the old `map add`/`map remove`; only the CLI word changed. Dropping the `map` noun and adding `bind` makes the reserved words `config profile image podman alias runtime status run bind` (a profile literally named `bind` still launches via `clause run bind`; `map` is a launchable profile name again)
- **`map show` folded into `status`, `map list` into `profile list`**: the two read-only `map` verbs were merged into the commands that already overlapped them, so the `map` noun could be retired whole. `status` (`cmd_status`) already printed the same `mapping:` and `mount:` lines `map show` did, so `cmd_map_show` and the `map-show` COMMAND were deleted outright (no code moved). `profile list` (`cmd_profile_list`) now prints a `profiles:` section (all profiles, marking nested ones) followed by a `mappings:` section (the full `~/.clause/clause.conf`, the old `map list`/`cmd_list_all` body), so `cmd_list_all` and the `list-all` COMMAND were removed. Both stay read-only: nullglob yields nothing when there is no profiles dir, and `-s` is false for a missing `clause.conf`, so neither creates `~/.clause`
- **`status` config dashboard**: `cmd_status` prints the full effective config for the current directory: profile, mapping, mount (+ source), effective args (+ source), effective effort (+ source), runtime, and image-built status. Read-only: in the `bootstrap_state` allowlist (guard is now `status|profile-list`, plus a `config` arm that is read-only only for `get`/`list`), so it never creates `~/.clause`. Its `mapping:` and `mount:` lines make `status` a superset of the removed `map show`. `config --list [--show-origin]` is the config-only sibling (the `git config --list` analog); `status` is the broader dashboard. Resolves the profile via `profile_for_management` (not `require_profile`), so a missing profile degrades to `(not created)` instead of erroring (`resolve_args`/`resolve_effort` guard each read with `[[ -f ]]`, so an uncreated profile dir just yields no profile-tier value). Runtime/image use a soft probe local to `cmd_status` (read `RUNTIME_FILE`, else `command -v podman`/`docker`, else `(none found)`), deliberately bypassing the hard `detect_runtime` (which exits when no runtime is present) so status works on a host with no container runtime, hence NOT added to `detect_runtime`'s allowlist. Now a bare subcommand word (`status`), special-cased in the `parse_args` loop and dispatched like the other nouns, not a flag
- **Per-profile `.gitconfig`** â€” each profile has its own `.gitconfig` bind-mounted at `/home/claude/.gitconfig`; starts empty, persists across sessions
- **Encoded workspace mount** â€” the host workspace is bind-mounted at `/workspace/<encoded-host-path>` and `-w` sets the container cwd to the same path, where `<encoded-host-path>` is the host workspace with `/` and `.` replaced by `-` (same scheme Claude uses for `~/.claude/projects/`). This keeps Claude's per-project state (logs, todos, history) separated by host workspace even when multiple workspaces share a profile. The encoding is factored into the `encode_path` helper (same two substitutions, same order â†’ default is byte-identical), and the *logical* path it encodes is resolved by `resolve_mount_path` so it can be pinned (see the mount override decision) to survive moving the host folder
- **Mount path override, layered (workspace + one-shot, no profile tier)**: a workspace can pin the container-side mount path so Claude's path-keyed history survives moving the folder on the host. `resolve_mount_path` sets `MOUNT_VALUE` (the logical host path that `encode_path` turns into `CONTAINER_WORKSPACE`, the bind target + cwd) and `MOUNT_SOURCE`, with precedence: (1) `--mount <path>` one-shot; (2) `<workspace>/.clause-mount` (first line; `read -r` trims surrounding whitespace, keeps internal spaces; empty â†’ ignored); (3) the real `$WORKSPACE` (default, unchanged). The bind-mount *source* is always the real `$WORKSPACE`; only the container-side target/cwd are pinned. The `.clause-mount` file lives inside the workspace deliberately, so it travels with the folder when moved (the mechanism that keeps the pin in effect); the stored value is a logical absolute path, kept verbatim (dumb/predictable). `validate_mount_path` requires a leading `/` and rejects a trailing slash except root `/` (a trailing slash encodes to a stray `-` that no realpath produces, silently orphaning history); so pin the canonical path (`$(pwd -P)`); the one-shot is validated at parse time (hard error), the file value is revalidated in `resolve_mount_path` (invalid â†’ warn on stderr + fall through to `$WORKSPACE`, so a hand-edit cannot break launch). `resolve_mount_path`'s early returns are `return 0` (load-bearing under `set -e`: a bare `return` would propagate the failed `[[ -f ]]` test and abort the bare call on the common no-override path). The override changes container *layout*, not `claude` args, so it applies in both claude and `-t` bash modes (called outside the `CONTAINER_CMD` branch in `cmd_launch`, unlike args/effort). Set/clear via `config mount <path>` / `config --unset mount` share `write_layered_file` / `remove_layered_file` with the args and effort keys; there is no profile tier, so `config --profile mount` is a parse-time error (workspace-scoped only). Nothing is seeded in `default/` and no override file exists by default, so launch is byte-identical until an override is set; `config` set/unset need no runtime and are not read-only, so they pass through the existing `bootstrap_state`/`detect_runtime` guards unchanged. `status` and `config --list` report the effective `mount:` path and its source (mount is not a `claude` arg, so it never enters the args line)
- **Git-style `config` subcommand (Tier A)**: the three layered knobs (args/effort/mount) are managed through one git-shaped subcommand instead of per-knob flags. Grammar: `clause [profile] config [--profile|--workspace] [--get|--unset|--list] [--show-origin] <key> [value]`, parsed by `parse_config_args` (Phase 1 consumes leading flags, stops at the first non-flag; Phase 2 takes a positional `<key> [value]` verbatim, so a dash-prefixed value like `config args '--effort high'` is never read as a flag). Keys are `args|effort|mount` (`validate_config_key`); value validation reuses `validate_effort` / `validate_mount_path` at parse time. Scope defaults to the workspace tier; `--profile` targets the profile tier (`--workspace` is the explicit default). `mount` is workspace-only, so `--profile mount` is a parse-time error. `config <key> <value>` sets, bare `config <key>` gets (implicit, git-style), `config --get` prints one raw effective value (scriptable, **no** effort-into-args injection, which stays a launch concern shown by `status`), `config --unset` clears, `config --list [--show-origin]` dumps all keys. `cmd_config` dispatches to `config_do_{set,unset,get,list}`, reusing the same `write_layered_file` / `remove_layered_file` / `resolve_args` / `resolve_effort` / `resolve_mount_path` helpers as launch, so precedence (workspace > profile) is identical; profile-scope writes call `require_profile`, reads use `config_resolve_profile_silent` (degrades on a missing profile like `cmd_status`). `bootstrap_state` treats `config` as read-only **only** for `get`/`list` (never creates `~/.clause`); `set`/`unset` bootstrap like the old setters. Never needs a runtime. This **hard-replaced 12 flags with no aliases** (consistent with the noun-verb clean-break policy): `-A/--args-view`, `--args-set{,-profile}`, `--args-remove{,-profile}`, `--effort-set{,-profile}`, `--effort-remove{,-profile}`, `--mount-set`, `--mount-remove` all now error as unknown options. `config` is a **reserved word** (a profile cannot be named `config`, the one cost of the subcommand). Unchanged: `status` (the broad dashboard, the `git status` analog) and the one-shot launch flags `-a`/`-e`/`--mount` (git keeps `-c` alongside dedicated flags); `config --list` is the `git config --list` analog. Scope naming `--profile`/`--workspace` was chosen (by the user) over git-literal `--global`/`--local` because clause's tier is per-profile, not one user-wide file
- **Noun-verb subcommands (Tier B)**: extending Tier A's `config`, every remaining management and info command became a git-style noun-verb subcommand, retiring the last management/info flags. Full surface: `config` (Tier A), `profile create|delete|list`, `image build|reset|suggest`, `podman enable|disable|reset`, `alias create|delete`, `runtime set <podman|docker>|unset`, `status`, the top-level `bind` / `bind --unset`, plus `run <profile>`. The words `config profile image podman alias runtime status run bind` are reserved (a bare word matching one is the subcommand, not a profile to launch); `run <profile>` is the escape hatch for launching a profile whose name collides with a command word (`clause image` runs the image subcommand, `clause run image` launches a profile named `image`, `clause image build image` manages that profile via the trailing profile arg). Clean break, no flag aliases: the old `-s`, `-l`, `-L`, `-C`, `-D`, `-R`, `-S`, `-b`, `-P`, `-m`, `-u` short flags and their `--status`, `--list-all`, `--profile-create`, `--profile-delete`, `--containerfile-reset`, `--containerfile-suggest`, `--build`, `--podman-enable`, `--podman-disable`, `--podman-reset`, `--alias-create`, `--alias-delete`, `--runtime-set`, `--runtime-remove`, `--map`, `--unmap` long forms all now error as unknown options. Parser design: `parse_subcommand` maps each `<noun>.<verb>` to the SAME internal `COMMAND` value the old flags used (`image build`â†’`build`, `image reset`â†’`reset-containerfile`, `profile create`â†’`create-profile`, `runtime unset`â†’`runtime-remove`, and so on), so the `main` dispatch `case` and every `cmd_*` body are reused unchanged; `set_command` now takes a multi-word label (e.g. `image build`) purely for the conflict-error message. `status` and `run` are special-cased in the main `parse_args` loop rather than `parse_subcommand`: `status` sets its command, `run` consumes the next token as a verbatim profile name (so a reserved-word name still launches) then stops. The old combined list view was replaced by `profile list` (COMMAND `profile-list` â†’ `cmd_profile_list`, listing profiles) and `status` (mapping + mount); the old `cmd_list` was removed. (When the `map` noun was later retired, `map show` folded into `status` and `map list`/`cmd_list_all` into `profile list`, which grew a `mappings:` section; see the `bind` bullet.) Grammar: session options (`-t -w -a -e --mount -y -n`) must precede the subcommand; a trailing token after `<noun> <verb>` is the optional profile (`clause work image build` == `clause image build work`), except `runtime set` which takes a required `podman|docker` value; a stray dash-token after a subcommand is a clear error; a `-h`/`--help` verb prints usage (like `config`). `bootstrap_state`'s read-only allowlist is now `status|profile-list` (plus `config` get/list); `detect_runtime`'s allowlist is unchanged (`build|delete-profile|podman-reset|launch`). Tier A's `config` established this pattern
- **`default/` â€” profile template** â€” the repo's `default/` directory mirrors a profile's layout (`Containerfile`, `clause-args`, `.claude/settings.json`, `.claude/CLAUDE.md`, `.claude/hooks/set-bg.sh`); `seed_profile` copies each file to the matching profile path on first use if missing, never overwriting afterward (users can freely modify their profile's copies). Empty runtime files (`.claude.json`, `.gitconfig`, `.claude/clause-sudo.log`) are scaffolded by the script, not tracked in `default/`. Seeded copies are deleted only when the profile is deleted
- **`bypassPermissions` by default** â€” `default/.claude/settings.json` ships with `permissions.defaultMode = "bypassPermissions"`. Profiles created before this change keep whatever was already in their `settings.json`.
- **Official plugins enabled by default**: `default/.claude/settings.json` ships with an `enabledPlugins` map turning on two official plugins for every new profile, `skill-creator@claude-plugins-official` and `claude-md-management@claude-plugins-official`. The `claude-plugins-official` marketplace is baked into the CLI and auto-installs on launch (unless `CLAUDE_CODE_DISABLE_OFFICIAL_MARKETPLACE_AUTOINSTALL` is set), so no marketplace registration or plugin-cache seeding is needed; the plugins install on the profile's first session, which needs network access. Profiles created before this change keep whatever was already in their `settings.json`.
- **`effortLevel` defaults to xhigh**: `default/.claude/settings.json` ships with `effortLevel: "xhigh"`, Claude Code's settings key for startup reasoning effort (valid values `low`, `medium`, `high`, `xhigh`; the launch resolver reads `settings.effortLevel` only when no `--effort` flag is passed). Because normal launches pass `--effort` from `clause-args` and that flag overrides the setting, `effortLevel` takes effect only for a bare `claude` run inside a `-t` terminal session. Deliberately kept in `settings.json` rather than raising the `clause-args` effort, so the launch default is unchanged. To override effort on a normal launch, use the layered effort override (the `resolve_effort` / `config effort` decision above), which injects `--effort` into the args; this `settings.json` value still governs only `-t` bare `claude` runs. Profiles created before this change keep whatever was already in their `settings.json`.
- **Nested podman is opt-in per profile**: `podman enable` creates the `~/.clause/profiles/<name>/nested` marker file and offers to append a managed block (between `# clause-nested-begin` / `# clause-nested-end`) to the profile's `Containerfile`; `podman disable` reverses both. The block's heredoc lives in the `clause` script, not in `default/`, so non-nested images stay lean and legacy profiles work the same way. Rebuild required after enabling.
- **Nested launch flags per runtime**: when the marker is present, launch adds `--device /dev/fuse --device /dev/net/tun --security-opt label=disable` plus the storage volume; docker additionally gets `seccomp=unconfined` and `apparmor=unconfined` (its default profiles block unshare/mount). Missing devices are skipped with a warning, and launch warns (non-fatally) if the profile `Containerfile` does not appear to install podman.
- **Per-profile nested storage volume**: inner podman storage lives in the named volume `clause-<profile>-containers` mounted at `/home/claude/.local/share/containers` (persists inner images across ephemeral sessions, allows native overlayfs, keeps nested-subuid-owned files out of the profile dir where `rm -rf` could not delete them). Removed automatically by `profile delete` and on demand by `podman reset` (typed confirmation); selective cleanup is `podman system prune` inside a session.
- **lazydocker in nested block**: the managed nested-podman block also installs the latest lazydocker release binary (same GitHub-redirect fetch as lazygit) and bakes `~/.config/lazydocker/config.yml` mapping compose actions to `podman-compose`. A `lazydocker` bash function (alias `ld`) appended to the container `.bashrc` starts `podman system service` on demand and points `DOCKER_HOST` at the socket. Profiles with the old block need `podman disable`, `podman enable`, and a rebuild to pick it up.
- **In-container `clause` alias**: `default/Containerfile` appends a `clause` alias to the container user's `.bashrc` that runs `claude --effort max --dangerously-skip-permissions` (mirrors the seeded `clause-args` default; handy in `-t` terminal sessions). Baked into the image at build time: profiles with an existing `Containerfile` pick it up only after `image reset` (or adding the line manually) plus a rebuild, since seeding never overwrites a profile Containerfile.
- **lazygit in base image**: `default/Containerfile` installs the latest lazygit release binary into `/usr/local/bin` (arch-aware, fetched from GitHub at build time by following the `releases/latest` redirect) and adds an `lg` alias to the container user's `.bashrc`. Same caveat as the `clause` alias: existing profile Containerfiles pick it up only after `image reset` (or manual edit) plus a rebuild.
